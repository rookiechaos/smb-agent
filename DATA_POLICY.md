# DATA_POLICY.md

This document formalizes the data handling baseline for commercial SMB use.
It complements [`SECURITY.md`](SECURITY.md), [`GOVERNANCE.md`](GOVERNANCE.md),
[`LEGAL_READINESS.md`](LEGAL_READINESS.md),
[`JAPAN_TRUST_READINESS.md`](JAPAN_TRUST_READINESS.md), and
[`RUNBOOK.md`](RUNBOOK.md).

## Principle

Customer data is operational data, not training exhaust.

Use it to deliver and improve the customer's service. Do not export it for
model training, routing-model development, demos, or sales material unless that
use is explicitly covered by the customer agreement.

## Local-only runtime artifacts

Live workspace and analytics artifacts stay on the dedicated operator machine.
That includes files such as `workspaces/<customer_id>/auth.json`,
`monitor_auth.json`, `employee_auth.json`, `operator_approvals.jsonl`,
`workflow_health.json`, `maintenance_report.json`, `workspaces/dashboard.html`,
and `analytics/memory_analytics.json`. They are operational records, not source
assets, and should not be committed back into the product repo except as
intentionally redacted samples.

## Data Classes

| Data | Examples | Default handling |
| --- | --- | --- |
| Company context | mission, vision, values, strategy, priorities, decision style, risk tolerance | Keep current snapshot; review when strategy changes |
| Raw voice audio | customer or employee speech used for intake | Local MLX ASR by default when voice is enabled; avoid retaining raw audio unless explicitly needed |
| Requirements | `requirements.json`, `transcript.txt` | Treat as customer confidential data |
| Voice transcripts | `transcript.txt`, copied requirements text, summaries | Confidential operational data; may include personal or sensitive information |
| Runtime logs | `runtime/chat.jsonl` | Store shape only: message length, skill, latency, error |
| Model usage ledger | `usage.jsonl` | Store provider, surface, stage, model, token counts when available, status, and timing metadata; no prompts or model outputs |
| Memory compaction ledger | `memory_compaction.jsonl` | Store rolling-summary, packed-context, and retrieval-hit metadata; no raw prompts or hidden reasoning |
| Failure memory | `failure_memory.jsonl` | Use for internal reliability analysis only unless customer terms allow broader use |
| Loop memory | `loop_memory.jsonl` | Adaptive loop metadata: budget, rounds used, complexity, benchmark policy; no raw prompts or reasoning |
| Run logs | `runs/**/*.log` | Secret-redacted, operator-only debugging records |
| Integration config | SMTP, calendar, CRM credentials | Operator-owned secrets; never generated by AI |

## Default Retention

The code-level defaults are:

| Record | Env var | Default |
| --- | --- | --- |
| General run/audit records | `SMBAGENT_DATA_RETENTION_DAYS` | 180 days |
| Runtime chat metadata | `SMBAGENT_RUNTIME_LOG_RETENTION_DAYS` | 90 days |
| Model usage ledger | `SMBAGENT_DATA_RETENTION_DAYS` | 180 days |
| Failure memory | `SMBAGENT_FAILURE_MEMORY_RETENTION_DAYS` | 365 days |
| Loop memory | `SMBAGENT_FAILURE_MEMORY_RETENTION_DAYS` | 365 days |
| Raw negotiation transcript | `SMBAGENT_TRANSCRIPT_RETENTION_DAYS` | 30 days |

Use:

```bash
smbagent retention-plan <customer_id>
```

This reports the active policy and expired records. It does not delete files.
Deletion remains an operator-owned action under [`GOVERNANCE.md`](GOVERNANCE.md).

## Training / Tuning Use

Failure memory and loop memory can become useful for future superparameter
tuning, adaptive budget selection, and routing models, but they must stay
governed.

Default:

```bash
SMBAGENT_ALLOW_FAILURE_MEMORY_TRAINING_USE=false
```

Set it to `true` only after customer-contract review. Even then, export
structured labels and operational metadata before raw transcripts or free text.

## Voice And Transcript Handling

Default local ASR reduces third-party exposure for raw audio. It does not
automatically protect transcripts or downstream requirements. Unless a separate
sensitive/local-only workflow is configured, transcript-derived content may flow
into API-backed or vendor-CLI stages such as Negotiation, Plan, Coding, and
Validation.

Operational rules:

- give recording/transcription notice before collecting voice input
- prefer `SMBAGENT_ASR_BACKEND=mlx` for Japan SMB pilots and sensitive work
- keep `SMBAGENT_ASR_DELETE_AUDIO_AFTER_TRANSCRIBE=true` unless the customer
  contract explicitly requires retained audio
- set `SMBAGENT_SENSITIVE_MODE=true` for clinics, payroll, GPS, employee
  monitoring, or other high-sensitivity workflows
- use `SMBAGENT_ASR_BACKEND=none` when text-only intake is safer
- use `SMBAGENT_ASR_BACKEND=api` only when customer terms allow cloud ASR
- keep transcript retention short; the default is 30 days
- do not include passwords, API keys, raw patient records, disciplinary details,
  or unnecessary employee-identifying data in voice intake
- for clinics, payroll, GPS, and employee-monitoring use cases, complete
  [`LEGAL_READINESS.md`](LEGAL_READINESS.md) before real-data processing

## Japan SMB Compliance Baseline

For Japan-facing SMBs, review the Personal Information Protection Commission
guidance and APPI obligations before launch:

- https://www.ppc.go.jp/en/legal/

This repo is not a legal compliance product. The operator must define:

- purpose of use
- data categories collected
- retention window
- access control
- incident contact
- whether data may be used for tuning or training

For GPS or employee-related workflows, also define employee-facing purpose,
access scope, retention, and HITL rules before ingesting real data.

For Japan SMB sensitive workflows, copy the workspace templates before launch:

```bash
smbagent japan-trust-note <customer_id>
```

The templates create:

- `japan_trust_launch_note.md`
- `customer_ai_use_policy_ja.md`
- `employee_data_notice_ja.md`
- `gps_analysis_notice_ja.md`

The local `smbagent trust-eval <customer_id>` command checks for these records
when the workspace context or requirements mention GPS, employee, payroll, or
clinic-sensitive signals. These records are not legal approval by themselves;
they are the minimum evidence that purpose, access, retention, notice, and HITL
boundaries were considered.
